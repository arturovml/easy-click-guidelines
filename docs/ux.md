# Easy Click — UX Manifesto
**Operational clarity. Zero friction. Confident action.**

## 1. Principio rector (North Star)

> Easy Click existe para que una persona pueda operar sin pensar.

La interfaz nunca debe exigir interpretación, memoria o aprendizaje previo.  
El sistema debe acompañar la acción, no interrumpirla.

---

## 2. El usuario de Easy Click

Easy Click no diseña para:
- Power users
- Perfiles técnicos
- Dashboards “bonitos”

Diseña para:
- Personas operando bajo presión
- Contextos físicos (mostrador, piso, calle)
- Decisiones rápidas y repetitivas
- Baja tolerancia al error

**Si dudas entre “más claro” o “más elegante”, siempre gana lo claro.**

---

## 3. Principios UX fundamentales

### 3.1 Claridad antes que densidad
- Mostrar solo lo necesario para la acción actual
- Información secundaria:
  - se oculta
  - se difiere
  - se agrupa

---

### 3.2 Una acción principal por pantalla
Cada vista debe responder:
> ¿Qué se espera que el usuario haga aquí?

- Un CTA principal
- Acciones secundarias subordinadas
- Nunca dos CTAs compitiendo

---

### 3.3 Estados visibles siempre
Toda acción tiene estado:
- idle
- loading
- success
- error
- empty

Nunca:
- loaders infinitos
- pantallas en blanco
- silencios tras un click

---

### 3.4 Feedback inmediato
Toda acción genera respuesta:
- visual
- espacial
- temporal

Si algo tarda, se comunica.

---

## 4. Jerarquía visual

### 4.1 Lo importante es grande
Totales, cantidades y estados críticos se ven, no se descubren.

### 4.2 Lo operativo es neutral
Inputs, listas y contenedores no compiten visualmente.

### 4.3 El color es semántico
- Naranja → acción primaria
- Verde → éxito
- Rojo → error / riesgo
- Neutrales → estructura

Nunca usar color solo por estética.

---

## 5. Ritmo y espaciado

Easy Click respira:
- Espaciado consistente
- Grillas simples
- Alineaciones claras

Debe sentirse estable y predecible.

---

## 6. Inputs y formularios

### 6.1 Menos es más
- Eliminar campos innecesarios
- Defaults inteligentes
- Autocompletar siempre que sea posible

### 6.2 Validación humana
- Mensajes claros
- Sin jerga técnica
- Indicar qué pasó y qué hacer

---

## 7. Errores ≠ fallos
Un error es parte del flujo:
- Nunca culpar al usuario
- Siempre ofrecer salida
- Mantener contexto

---

## 8. Accesibilidad no negociable
- Focus visible
- Targets grandes
- Buen contraste
- Navegación por teclado

---

## 9. Consistencia entre productos
Easy POS, Parking y Production deben sentirse iguales.  
Aprendes uno, entiendes los demás.

---

## 10. Regla final

> Si algo necesita explicación, el diseño falló.
